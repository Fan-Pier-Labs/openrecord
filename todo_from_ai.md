The biggest improvement you can make to the MyChart MCP is server-side response trimming. Right now, every endpoint returns the raw MyChart JSON, which is packed with encrypted keys, nested HTML/CSS report content, historical result arrays, empty metadata fields, and UI-specific flags that an AI consumer will never use. The MyChart billing response alone was so large it overflowed the context window. For each endpoint, you should strip the response down to only the fields an AI would act on. Lab results should return order name, date, status, and for each component just the name, value, units, reference range, and abnormal flag — not the full `historicalResults`, `reportDetails`, `studyResult`, `scans`, `imageStudies`, or encrypted `componentID` keys. Billing should return date, description, provider name, payer name, charged amount, insurance paid, self-paid, deductible/copay/coinsurance breakdown, balance due, and status — not the full `ProcedureList` payment sub-objects, `CoverageInfoList` when it's redundant with the visit-level summary, or `AgencyInformation`. Messages should return subject, date, author name, and body as plain text with HTML stripped — not the raw HTML divs with inline styles, encrypted `wmgId`/`wprKey` values, or the full `userOverrideNames`/`viewers` maps. Imaging should return order name, date, provider, and the impression text extracted from the HTML — not the entire `reportContent` HTML blob with its embedded CSS stylesheets. Linked accounts should drop the logo URLs. A quick `format=summary` vs `format=full` query parameter would let you support both AI and UI consumers from the same endpoints.

Beyond trimming, you should add pagination with a default limit (say 10 items) and `limit`/`offset` parameters on labs, messages, and billing. The MyChart account returned 45 billing line items and massive lab/imaging/message payloads in a single call — most AI queries only need the most recent few. You should also fix the `get_insurance` endpoint, which returned `hasCoverages: true` with an empty `coverages` array on both instances — the actual plan name, group number, and deductible/OOP max limits are probably behind a separate page load that the scraper isn't hitting. That's a high-value data point for exactly the kind of analysis we just did (plan comparison, deductible tracking). On error handling, the MCP server should return descriptive error messages (HTTP status, brief reason, maybe a server-side request ID) instead of letting Anthropic's transport layer swallow everything into a generic "Error occurred during tool execution" — that made debugging the outage impossible from the client side. Finally, flatten the deeply nested structures where you can: instead of `resultComponents[].componentResultInfo.referenceRange.formattedReferenceRange`, a flat `{ name, value, units, range, abnormal }` object is far more token-efficient and easier for an AI to reason over. These changes combined would probably cut response sizes by 80-90% and make the connector significantly more useful as an AI-native data source.
